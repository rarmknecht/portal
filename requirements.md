## 1. Overview

Portal is a two-component personal media system. An **agent** runs on a desktop or server machine and exposes locally stored media. A **mobile app** discovers the agent over the local network (or Tailscale), browses the exposed media, and streams playback to the phone.

The design goal for v1 is a **polished single-user tool**: the experience should feel intentional and reliable, not like a prototype, but complexity is deliberately constrained. No cloud, no accounts, no multi-user.

**Analogy:** Think of Portal as a personal Plex, but simpler — a private hallway between one desktop and one phone, with no lobby, no ticket booth, no advertising board.

---

## 2. Assumptions and scope

### In scope for v1
- One agent, one user, one or more of that user's mobile devices.
- Video, audio, and photos.
- Streaming playback on the phone (not casting, not downloading).
- Local network (LAN) and Tailscale as transport.
- Trusted-network security model.

### Out of scope for v1
- Multi-user or multi-tenant operation.
- Internet-exposed deployment outside Tailscale.
- Transcoding between codecs (pass-through only; see §6).
- Offline/download-for-later playback.
- Casting to Chromecast, AirPlay, or other receivers.
- Uploads from phone to agent.
- Editing, deletion, or any write operations on the agent's files.

### Platforms
- **Agent:** Windows, macOS, Linux.
- **Mobile app:** iOS and Android.

## 3. User experience requirements

### 3.1 First-run setup

**Agent (desktop):**
1. User installs and launches the agent.
2. Agent presents a minimal configuration UI (system tray / menu bar / web UI — TBD, see §8).
3. User selects one or more folders to add to the **allowlist**. Nothing outside the allowlist is ever visible to the mobile app.
4. Agent displays its connection details: hostname, IP, port, and a QR code that encodes the pairing info.
5. Agent shows a clear status indicator: running, idle, or serving a client.

**Mobile app:**
1. User installs and launches the app.
2. App scans for agents on the local network (mDNS/Bonjour) and displays any found.
3. If none are found, user can enter connection details manually or scan the QR code from the agent.
4. On first connection, app remembers the agent for future sessions.

### 3.2 Browsing

- The app presents allowlisted folders as top-level "libraries."
- Within a library, the user can navigate the folder tree naturally.
- Each item shows a thumbnail (generated by the agent), name, type, and duration or size.
- Sort options: name, date modified, size, duration.
- Search within the current library by filename.

### 3.3 Playback

- Tapping a media item begins streaming within 2 seconds on a typical LAN.
- Video player supports standard controls: play/pause, scrub, volume, fullscreen, picture-in-picture (where the OS allows), and playback speed.
- Audio player supports standard controls and continues playing when the app is backgrounded.
- Photo viewer supports pinch-zoom, swipe between photos in the same folder, and basic EXIF display (date taken, dimensions).

### 3.4 Reliability expectations

- If the network drops mid-playback, the app shows a clear "reconnecting" state and resumes at the current timestamp when the agent is reachable again.
- If the agent is stopped, the app says so clearly rather than hanging.
- If a media file is in an unsupported codec, the app says "this format isn't supported" rather than silently failing.

### 3.5 Polish bar

These are the details that distinguish "polished" from "MVP":
- Smooth thumbnail loading with placeholders, not flashing blanks.
- Sensible defaults: remember last library, last sort order, last volume.
- Native platform feel — iOS should feel like an iOS app, Android like an Android app.
- No modal dialogs for recoverable errors; use inline banners.
- Dark mode on both platforms.

## 4. Technical architecture

### 4.1 Components

```
┌─────────────────────┐        ┌─────────────────────┐
│   Mobile App        │        │   Agent             │
│   (iOS / Android)   │ ◄────► │   (Win/macOS/Linux) │
│                     │  LAN   │                     │
│   - Discovery       │  or    │   - Media indexer   │
│   - Browse UI       │  TS    │   - HTTP API        │
│   - Media players   │        │   - Streaming       │
│                     │        │   - Thumbnailer     │
└─────────────────────┘        └─────────────────────┘
```

### 4.2 Agent

**Runtime:** Python 3.11+ is the natural choice given your preference and the ecosystem fit (FFmpeg bindings, filesystem libraries, and HTTP frameworks are all mature).

**Core responsibilities:**
- Read-only scan of allowlisted paths, with background indexing on startup and incremental updates via filesystem watches.
- Maintain a lightweight index (SQLite) of discovered media: path, type, size, duration, codec, last-modified.
- Serve an HTTP API for discovery, browsing, metadata, and streaming (for the mobile app).
- Serve a local web UI on `http://localhost:<port>` for configuration and status (for the user on the agent machine).
- Generate and cache thumbnails, preferring existing embedded thumbnails when available, regenerating otherwise.
- Advertise itself via mDNS/Bonjour as `_portal._tcp.local`.

**Recommended libraries:**
- **FastAPI** for the HTTP API — modern, async, good docs.
- **Uvicorn** as the ASGI server.
- **FFmpeg** (system binary) + **ffmpeg-python** for metadata extraction and thumbnail generation.
- **zeroconf** for mDNS advertisement.
- **watchdog** for filesystem change detection.
- **SQLite** (standard library) for the index.

**HTTP API surface (v1):**

| Method | Endpoint | Purpose |
|--------|----------|---------|
| GET | `/api/v1/health` | Liveness check |
| GET | `/api/v1/libraries` | List allowlisted root folders |
| GET | `/api/v1/browse?path=...` | List children of a path within an allowlisted root |
| GET | `/api/v1/metadata?path=...` | Return metadata for a single item |
| GET | `/api/v1/thumbnail?path=...&size=...` | Return a cached thumbnail |
| GET | `/api/v1/stream?path=...` | Stream media with HTTP Range support |
| GET | `/api/v1/search?library=...&q=...` | Filename search within a library |

All paths are validated against the allowlist on every request. Path-traversal attempts (`..`, symlinks pointing outside the allowlist) must be rejected.

### 4.3 Mobile app

**Framework:** Flutter. One codebase for iOS and Android, mature video player plugins, and the native feel we need for a polished single-user tool is achievable. Underlying playback still uses the platform-native players (AVPlayer on iOS, ExoPlayer on Android) via Flutter plugins.

**Responsibilities:**
- mDNS discovery (`_portal._tcp.local`) on the local network.
- Persist known agents and last-used paths locally.
- Render browse UI against the agent's API.
- Stream video via platform-native player (AVPlayer on iOS, ExoPlayer on Android).
- Stream audio via platform-native audio player with background playback.
- Display photos with a native image viewer.

### 4.4 Media handling

Agent streams media as-is using HTTP Range requests. The mobile player must natively support the codec or the file will not play.

**v1 expected-to-work matrix** (items the mobile players handle natively):
- **Video:** H.264 / H.265 in MP4 or MOV containers.
- **Audio:** AAC, MP3, FLAC.
- **Photos:** JPEG, PNG, HEIC (iOS native; Android needs library support).

Unsupported formats are listed in the browse UI with a "not playable" indicator rather than being hidden, so the user knows the file exists.

### 4.5 Discovery and transport

- **Primary discovery:** mDNS/Bonjour advertisement by the agent.
- **Fallback:** manual entry of `host:port` or QR-code scan.
- **Transport:** plain HTTP over LAN. Tailscale provides the encrypted tunnel when used remotely, so the agent does not need to terminate TLS itself in v1. (This is a deliberate v1 simplification and is discussed in §5.)

### 4.6 Local web UI (agent)

The agent's configuration and status surface is a small web UI served by the agent itself. This avoids writing three native tray/menu-bar apps and keeps the stack uniform.

- **Binding:** `127.0.0.1` only by default for the web UI. The media API binds separately per §4.6 configuration.
- **Default port:** `5567`, configurable.
- **Capabilities:** manage the folder allowlist, view agent status (running, indexing, streams in progress), view/rotate logs, adjust thumbnail cache size, adjust log size cap, show the pairing QR code and discovery details.
- **Authentication:** none in v1, because it is bound to localhost on the agent machine. Documented as a trusted-loopback design choice, same model as §5.
- **Tech:** served by the same FastAPI process; HTML/CSS/vanilla JS is sufficient — no separate SPA framework needed for a config surface this small.

### 4.7 Configuration

Agent configuration lives in a single TOML file. Defaults are chosen so the agent runs with no edits; the web UI writes this file when the user changes settings.

```toml
[agent]
media_api_bind = "0.0.0.0"
media_api_port = 7842
web_ui_bind = "127.0.0.1"
web_ui_port = 5567

[libraries]
allowlist = [
    "/home/user/Videos",
    "/home/user/Music",
    "/home/user/Pictures"
]

[indexing]
mode = "background"          # full scan on startup, then filesystem-watch updates
scan_on_startup = true

[thumbnails]
cache_dir = "~/.portal/thumbnails"
max_cache_size_mb = 500
prefer_embedded = true       # use existing embedded thumbnails when present

[logging]
log_dir = "~/.portal/logs"
max_size_mb = 150            # total cap across rotated files
rotation = "size"
```

## 5. Security requirements

The v1 security model is **"trusted network."** This is a deliberate scoping decision, not an oversight. Every design choice below flows from that model.

### 5.1 Threat model (v1)

**In scope to defend against:**
- Accidental exposure of files outside the allowlist (path traversal).
- Agent crashing or being exploited via malformed HTTP input.
- Malformed media files causing thumbnail generation or metadata extraction to crash the agent.

**Explicitly out of scope:**
- Hostile users on the same LAN. The user is expected to run Portal on a network they control, or to use Tailscale for remote access.
- Internet exposure. The agent must not be published to the public internet without Tailscale or equivalent in front of it. The documentation must state this.

### 5.2 Requirements

- Allowlist enforcement: every path parameter in the API is canonicalized and checked against the allowlist roots. Symlinks pointing outside the allowlist are rejected.
- Read-only: the agent does not expose any mutating endpoints.
- Bind address: default to `0.0.0.0` but make it a documented, configurable setting for users who want to bind only to the Tailscale interface.
- Input validation: all query parameters are length-limited and type-checked; FFmpeg and metadata calls run with a timeout.
- Logging: structured logs including request path, source IP, and status, retained locally. No telemetry leaves the agent.

### 5.3 Documented guidance to the user

The agent's documentation must clearly state:
- Portal is designed for trusted networks.
- Do not expose the agent port on the public internet.
- For remote access, use Tailscale (or equivalent) to create an authenticated, encrypted tunnel.

### 5.4 Path to a stronger model (post-v1)

Called out here so v1 choices do not paint us into a corner: a future version should add device pairing with a shared secret, TLS termination on the agent with a self-signed cert the app pins on first pairing, and per-request token auth.

## 6. Non-functional requirements

| Category | Target |
|----------|--------|
| Time to first frame (LAN) | ≤ 2 seconds for supported codecs |
| Browse response time | ≤ 500 ms for folders up to 1,000 items |
| Thumbnail generation | ≤ 3 seconds on first request, cached thereafter |
| Agent memory footprint | ≤ 300 MB at idle |
| Agent CPU at idle | Negligible (< 1% on modern hardware) |
| Concurrent streams | At least 2 simultaneous streams on LAN |

## 7. Resolved design decisions

The following decisions have been made for v1 and are reflected in sections 3 through 6 above. They are captured here for traceability.

| # | Decision area | Resolution |
|---|---------------|------------|
| 1 | Agent UI surface | Local web UI served by the agent, bound to `127.0.0.1`, default port `5567`, configurable. See §4.6. |
| 2 | Mobile framework | Flutter, with platform-native players used under the hood via plugins. See §4.3. |
| 3 | Photo thumbnails | Use existing embedded thumbnails when present; regenerate otherwise. Cached in the agent's thumbnails directory. See §4.7. |
| 4 | Indexing strategy | Background indexing. Full scan on agent startup, then incremental updates via filesystem watches. See §4.2. |
| 5 | Logging | Local log directory under the agent's data directory. Default total size cap of 150 MB across rotated files, configurable. See §4.7. |

## 8. Success criteria for v1

Portal v1 is done when:
- A user can install the agent on any of the three supported desktop OSes, point it at folders, and have the mobile app discover and browse those folders within 60 seconds of first launch.
- Streaming a supported video over LAN starts within 2 seconds and plays smoothly for the duration of the file.
- The user has never had to edit a config file unless they wanted to.
- The app feels like a native app on both iOS and Android, not a web wrapper.
- The user never sees an unexplained failure — every error state has a human-readable message and a sensible recovery path.
